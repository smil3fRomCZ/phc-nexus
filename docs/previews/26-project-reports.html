Datovy model

  • Zadne nove tabulky — vsechny reporty se pocitaji z existujicich dat (tasks, time_entries, epics, approvals, comments, project_members)
  • Kazdy widget = agregacni dotaz nad stavajicimi modely

Backend

  • ReportController s metodami pro kazdy widget, vracejici JSON
  • Kazda metoda respektuje date_range parameter (from/to timestamps)
  • Query scope: Project::reports()->burndown($from, $to)
  • Velke agregace (burndown, velocity) jako dedicated Actions: ComputeBurndown, ComputeVelocity

Cache strategie

  • Redis cache s 5-minutovym TTL pro computovane agregaty
  • Cache key: report:{project_id}:{widget}:{date_hash}
  • Invalidace: event listener na task/time_entry CRUD — clearuje relevatni keys

Frontend

  • Doporuceni: recharts (React-native, lehka, SSR-friendly)
  • Alternativa: nivo pro heatmapy
  • Inertia props pro initial load, TanStack Query pro lazy-load widgetu
  • Widget layout: responsive grid, 2 sloupce na desktopu, 1 na mobilu

Export

  • Print-friendly HTML (window.print() s @media print styly)
  • PDF export: budouci rozsireni (dompdf/puppeteer)
  • CSV export jednotlivych tabulek (jiz existujici infra)

Routy

  • GET /projects/{project}/reports — Inertia page
  • GET /projects/{project}/reports/burndown — JSON widget data
  • GET /projects/{project}/reports/velocity — JSON widget data
  • GET /projects/{project}/reports/workload — JSON widget data
  • GET /projects/{project}/reports/risks — JSON widget data
